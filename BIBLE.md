# BIBLE — rag-approval-blueprint

The stable mind of this repo: invariants and the decision register. Public-safe (no business
internals). Wins on any in-repo conflict. Change it deliberately, with a commit.

## Zone

Bridge / Tool — real open source. Code MIT, documents CC BY 4.0. Public repository.
See `dev/base/CONSTITUTION.md` §1.

## What this repo exists for

The one sentence that defines it: **Everyone shows you how to build compliant RAG. This repo
shows you how to get it approved.** The gap it fills is the path from a finished architecture to
formal sign-off in a BaFin-regulated house — 2nd line, information security, data protection,
internal audit, board submission. Not another RAG security pattern library.

## Invariants

- **INV-1 — Verified references only.** Every chapter/article/paper citation was checked against
  a source during the work and carries its retrieval date. Unverifiable claims are phrased
  generically and listed under "Offene Punkte". Enforced by review, not by script: treat any
  uncited regulatory number in a diff as a defect.
- **INV-2 — No client, employer, or institution internals.** All templates newly written from
  generic supervisory logic; all examples fictional. Enforced by `scripts/gate.sh`
  ("no customer-internal names") and by the internal briefing staying git-ignored.
- **INV-3 — `controls/controls.yaml` is the single source of truth.** The rendered Markdown
  catalogue and readiness report are generated by `tools/render_controls.py`, never hand-edited.
  A test fails if the committed rendering drifts from the YAML.
- **INV-4 — Every control carries all four mandatory fields**: Kontrollziel · Prüfhandlung ·
  Evidenz-Artefakt · Mapping. A control missing one of them fails the schema test.
- **INV-5 — Every document ends with "Offene Punkte" / "Open questions".**
- **INV-6 — "Not legal advice"** appears in `README.md`, every `akte/` template, and the mapping.
- **INV-7 — Pilot integrity.** Every file under `pilot/` is marked as a fictional institution.
  The documented conflict and the two red controls stay in; the pilot is never polished clean.
- **INV-8 — One tool only.** `tools/render_controls.py`, under 250 lines, ruff-clean, tested.
  No second tool, no framework, no plugin surface.

## Decision register

Newest first. Each: date · decision · why · (superseded by …).

- **2026-07-19 — BAIT/VAIT are treated as historical vocabulary, DORA as the binding source.**
  Verified during the build: VAIT, KAIT and ZAIT were repealed effective 16./17.01.2025 and BAIT
  is being phased out (fully by 31.12.2026) because DORA (Regulation (EU) 2022/2554) supersedes
  them. *Why:* the briefing asked for a BAIT/VAIT mapping; presenting a repealed circular as the
  binding requirement would be exactly the kind of wrong citation INV-1 exists to prevent. The
  mapping therefore leads with DORA and MaRisk and keeps BAIT/VAIT as the chapter vocabulary that
  German audit and 2nd-line functions still speak — stated with dates. The filename
  `docs/mapping-bait-vait-dora.md` is kept as the briefing specified, because that is what
  readers search for.
- **2026-07-19 — The pilot is a release condition for v0.1.0.** Without an end-to-end run the
  repo is a template dump; the worked case is the proof. Per briefing M7.
- **2026-07-19 — Documents in German, shell in English.** Freigabeakte, Löschnachweis and
  mapping are German because German supervisory practice is the gap being filled; README,
  control fields, and `docs/executive-summary.en.md` are English. The README states the language
  logic openly rather than apologising for it.
- **2026-07-19 — Repo onboarded to the `dev/base` paved road** via `base sync` (additive):
  backbone scripts, session skills, security configs, CI, canonical CLAUDE.md/AGENTS.md.
